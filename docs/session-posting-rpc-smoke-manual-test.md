# M-14D-15C 依頼書RPC smoke test 手動実行手順・テストデータ設計

## 目的

依頼書まわりの既存RPCについて、実際にsmoke testを行う前に、手動実行手順とテストデータ設計を整理する。

確認目的:

- `create_session_post(...)` / `update_session_post(...)` / `delete_session_post(text)` の実動作確認。
- 権限ロール別の成功 / 拒否確認。
- 入力バリデーション確認。
- 完全削除時のCASCADE影響確認。
- 画面、DOM、console、記録docsに内部識別子や認証系の生値が出ないことの確認。

この工程では手順整理のみで、SQL Editor実行、DB/RPC変更、フロント実装、実データ作成・更新・削除、Discord実送信、Edge Function deployは行わない。

## 推奨テスト方式

- まずは実ブラウザ操作を中心に確認する。
- SQL Editorでの直接RPC確認は、画面から確認できない拒否系や境界系に限定する。
- テスト用依頼書を明示的に作成し、その依頼書だけを使う。
- 既存本番寄りデータ、公開中の運用データ、静的JSON由来データは編集・削除対象にしない。
- 完全削除確認は、必ず削除専用のテスト依頼書で行う。
- 参加申請 / コメントのCASCADE確認を行う場合は、CASCADE確認専用のテスト依頼書を別に用意する。
- SQL Editorで確認する場合も、サーバ高権限を前提にした手順は使わない。
- フロントからDBを直接更新・削除する方針にはしない。操作は既存RPCまたは既存画面導線に限定する。

## テストデータ設計

基本テスト用依頼書:

- タイトル例: `RPC smoke test temporary session`
- 概要例: `RPC smoke test用。確認後に削除。`
- 公開状態: 初期は非公開寄りにする。
- 募集状態: 初期は下書きまたは募集中前の安全な状態にする。
- Discord通知: 未実装扱いのため、実送信は行わない。
- 公開IDなどの識別値: 結果記録時は一般名またはマスクした表現にする。

削除確認用依頼書:

- 基本確認用とは別に作る。
- 完全削除後にcalendar、session-detail、session-post管理対象から消えることを確認する。
- 中止として残す確認と完全削除確認を混ぜない。

CASCADE確認用依頼書:

- 参加申請 / コメントを含む削除影響確認が必要な場合だけ使う。
- 関連申請・コメントが削除されても問題ない専用データに限定する。
- CASCADE確認後は、結果に実値を残さず、削除済み対象は一般名で記録する。

## ロール別確認観点

| 文脈 | 作成 | 更新 | 完全削除 | 管理対象表示 / UI表示 | 期待 |
| --- | --- | --- | --- | --- | --- |
| 未ログイン | 不可 | 不可 | 不可 | GM/admin向け操作を表示しない、または実行不可 | ログイン要求や権限不足として安全に止まる |
| 通常PL | 不可または権限拒否 | 他人の依頼書は不可 | 他人の依頼書は不可 | GM/admin向け管理UIを表示しない | 画面とRPCの両方で管理操作が通らない |
| 作成者GM | テスト用依頼書を作成可 | 自分のSupabase由来依頼書を更新可 | 自分の削除専用依頼書を完全削除可 | 自分の管理対象として表示 | 既存RPC経由で作成・更新・削除できる |
| 他GM | 権限がなければ不可 | 他GMの依頼書は不可 | 他GMの依頼書は不可 | 他GMの管理対象として表示しない、または編集不可 | 他人の依頼書へ操作が通らない |
| admin | 作成可 | Supabase由来依頼書を横断管理可 | Supabase由来依頼書を横断管理可 | 管理対象として表示 | adminはアプリ内権限としてのみ扱う |
| 静的JSON由来 | DB対象外 | DB対象外 | DB対象外 | 編集不可 / 削除不可理由を表示 | 既存RPCへ流さない |

## バリデーション確認観点

| 観点 | 推奨確認方法 | 期待 |
| --- | --- | --- |
| 下書き状態の公開化 | ブラウザ操作中心 | 保存前またはRPCで拒否される |
| 不正な募集状態 | 必要なら最小限の直接RPC確認 | 一般化されたエラーになる |
| 不正な公開状態 | 必要なら最小限の直接RPC確認 | 一般化されたエラーになる |
| 募集人数の上下逆転 | ブラウザ操作中心 | 保存前またはRPCで拒否される |
| 終了日時が開始日時以前 | ブラウザ操作中心 | 保存前またはRPCで拒否される |
| タイトル空欄 | ブラウザ操作中心 | 保存前に止まる |
| 概要空欄 | ブラウザ操作中心 | 既存仕様に沿って止まる、または安全な扱いになる |
| エラー表示 | ブラウザとconsole確認 | 内部識別子や生のRPCエラーを表示しない |

## 削除 / CASCADE確認観点

- `delete_session_post(text)` は完全削除として扱う。
- 完全削除では、該当依頼書に紐づく参加申請と参加希望コメントもDB制約によりCASCADEされる。
- 削除前確認文に、参加申請とコメントも削除される旨が表示される。
- 完全削除後、calendar、session-detail、session-post管理対象に残らない。
- 中止として残す場合は `status = canceled` を使い、完全削除とは分ける。
- CASCADE確認は、専用のテスト依頼書と削除されてもよいテスト申請・コメントだけで行う。
- Discord投稿削除同期は未実装扱いのため、今回のsmoke testでは実送信・実削除確認を行わない。

## 静的JSON由来確認観点

- 静的JSON由来はDB RPC対象外。
- session-detailで編集不可 / 削除不可理由が表示される。
- 管理対象selectに静的JSON由来を混ぜない。
- 同じ公開IDのSupabase由来がある場合はSupabase側を優先する。
- Supabase側が非公開、下書き、中止の場合に静的JSON fallbackで復活しない。
- 静的JSON由来の確認はブラウザ表示・マージロジック側で行い、DB RPCへ流さない。

## 実行順案

1. 事前確認
   - 対象環境、確認アカウント種別、console確認方法を決める。
   - 既存本番寄りデータを触らないことを確認する。
2. 作成者GMで基本テスト用依頼書を作成
   - タイトルと概要はテスト用であることが分かる値にする。
   - 初期公開状態は安全側にする。
3. 作成者GMで更新
   - タイトル、概要、日時、募集人数、公開状態、募集状態を変更して保存する。
   - calendar / detail / 管理対象表示への反映を確認する。
4. バリデーション確認
   - 下書き公開、人数上下逆転、終了日時逆転、空欄などを確認する。
   - エラー表示に内部情報が出ないことを見る。
5. 通常PL / 他GMで拒否確認
   - 管理UIが出ない、または編集・削除できないことを確認する。
6. adminで管理確認
   - Supabase由来依頼書を管理対象として扱えることを確認する。
   - adminはアプリ内権限としてのみ扱う。
7. 削除専用テスト依頼書で完全削除確認
   - 削除前確認文を確認する。
   - 完全削除後に表示・管理対象から消えることを確認する。
8. 必要な場合だけCASCADE確認
   - 専用依頼書、専用申請、専用コメントで行う。
   - 削除後の影響を一般名で記録する。
9. 結果記録
   - 確認済み、失敗、保留、残課題を記録する。
10. cleanup確認
   - テスト用依頼書が残っていないこと、または残す場合の理由を記録する。

記録時の注意:

- 実際の内部識別子、認証系の生値、申請・PC選択関連の内部キーは書かない。
- テスト用依頼書の公開IDなどが必要な場合は、一般名またはマスク表現にする。
- console errorがある場合も、機微値を含む全文貼り付けは避け、要約する。
