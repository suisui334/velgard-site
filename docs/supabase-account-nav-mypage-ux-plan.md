# Supabase 共通アカウント導線 / マイページUX設計書

## 1. 目的

この設計書は、Supabase Authを本番サイトへ入れる前に、共通ヘッダーのアカウントアイコンとマイページ導線のUXを整理するためのものです。

目的:

- Supabase Auth導入時の共通アカウント導線を設計する
- `session-detail.html` 本文中にログイン状態を常時表示しない
- 共通ヘッダー付近にアカウントアイコンを置く
- クリックまたはタップでマイページへ遷移する導線を検討する
- ログイン済み / 未ログイン / 読み込み失敗時の表示を整理する
- 本番実装前にUI配置、表示情報、安全条件を固定する

この工程では本番実装、HTML / JS / CSS変更、Supabase追加SQL実行は行わない。

## 2. 現在のヘッダー前提

現在の共通ヘッダーは、各HTMLの `#site-header` に `assets/js/main.js` が描画している。

現行構造の要点:

- `renderHeader(site, page)` が `<header class="site-header">` を生成する
- `.header-inner` はブランドリンク、ハンバーガーボタン、グローバルナビを横並びにする
- PCでは `.global-nav` が右側に表示される
- 900px以下では `.nav-toggle` が表示され、`.global-nav.is-open` がドロップダウン表示になる
- `session-detail.html` 本文中のログイン状態常時表示はF-7aローカル確認後に採用しない方針へ変更済み

アカウント導線は、既存ナビゲーションと本文表示を邪魔しない小さな共通UIとして扱う。

## 3. 共通アカウントアイコン方針

配置候補:

| 候補 | 内容 | 評価 |
| --- | --- | --- |
| 右上 | PCではナビ末尾またはナビ右隣、スマホではハンバーガー付近に置く | もっとも自然。既存ヘッダーの視線導線と相性がよい |
| 左上 | ブランド付近に置く | ブランド表示と競合しやすい |
| ナビ末尾 | `CALENDAR` の後ろにアカウント項目を置く | 実装は単純だが、スマホの折りたたみメニュー内に入り、常時導線として弱い |

推奨案:

```text
PC: グローバルナビ右端に小さなアカウントアイコンを置く
スマホ: ヘッダー右側でハンバーガーボタンの隣にアカウントアイコンを置く
```

表示方針:

- ヘッダー上は原則アイコンのみ
- ログイン済み時は `display_name` の頭文字または汎用ユーザーアイコンを表示候補にする
- 未ログイン時も同じ場所に汎用アカウントアイコンを置き、マイページ / ログイン導線へ進める
- 長い `display_name` はヘッダーには常時表示しない
- `aria-label` やツールチップでは「アカウント」「マイページ」程度の短い説明に留める

判断基準:

- 既存ヘッダーを邪魔しない
- スマホで押しやすい
- PLがログイン状態を意識しすぎない
- 必要な時だけマイページへ行ける
- 世界観ページやギャラリーの閲覧体験を邪魔しない

## 4. マイページ導線案

候補ファイル:

```text
mypage.html
assets/js/renderMypage.js
assets/js/accountNav.js
assets/js/supabaseAuthClient.js
```

役割案:

| ファイル | 役割 |
| --- | --- |
| `mypage.html` | ログイン状態、ログイン / ログアウト、将来の自分用情報を表示するページ |
| `assets/js/renderMypage.js` | マイページ本文の描画 |
| `assets/js/accountNav.js` | 共通ヘッダー上のアカウントアイコン状態表示 |
| `assets/js/supabaseAuthClient.js` | Supabase Auth client初期化とAuth状態取得を隔離 |

マイページに表示する将来候補:

- `display_name`
- 自分の申請中セッション
- 参加確定セッション
- 待機中セッション
- 自分のコメント履歴
- ログアウトボタン

初期最小案:

```text
ログイン状態
display_name
ログインフォーム
ログアウト導線
「参加申請一覧は今後対応」表示
```

最初から申請一覧やコメント履歴まで入れると、必要RPC / 表示権限 / エラー処理の範囲が広がる。初期版はAuth状態とログイン / ログアウトに限定する。

## 5. 表示してよい情報 / 表示しない情報

表示してよい情報:

- ログイン中 / 未ログイン
- `display_name`
- 自分の申請状態
- 自分の参加予定セッション
- 簡潔なエラー文

表示しない情報:

- `user_id` 全文
- `discord_user_id`
- email
- access token
- refresh token
- service_role
- secret key
- DB password
- 内部role
- Supabase Project URL
- anon key / publishable key

メール表示は、本番PL向けでは原則出さない。必要になった場合も、マイページ内の限定表示として別途検討する。

## 6. ログイン方式・ログアウト方式

候補:

| 候補 | 内容 | 評価 |
| --- | --- | --- |
| マイページにログインフォーム | アカウントアイコンから `mypage.html` へ遷移し、そこでログインする | 推奨。本文ページを汚さず責務が明確 |
| ログイン専用ページ | `login.html` を分ける | 将来拡張にはよいが、初期導入ではページが増える |
| セッション詳細本文にフォーム | `session-detail.html` 内でログインする | F-7a方針に反するため採用しない |

推奨案:

```text
共通ヘッダーはアカウントアイコンのみ
マイページでログイン / ログアウトを扱う
session-detail.html本文中にはログインフォームを置かない
```

ログアウト後の遷移:

- 初期案では `mypage.html` に留まり、未ログイン状態を表示する
- 将来、操作元ページへ戻る導線が必要なら `returnTo` 相当のURL設計を検討する

Authセッション復元:

- F-3 devプロトタイプでAuth状態取得と `public_profiles.display_name` 取得は確認済み
- 本番ではログアウトしない限りログイン状態が維持される前提で設計する
- ただしURL / key / tokenを独自にlocalStorageへ保存しない
- Supabase Auth標準のセッション永続化を使う場合は、保存先とログアウト時の削除挙動を実装前に明文化する

## 7. Supabase接続情報の扱い

必須方針:

- service_roleは絶対に使わない
- secret key / DB passwordは絶対に使わない
- anon / publishable keyは公開前提だが、docsやチャットに実値を書かない
- Project URL / anon key実値はREADMEやdocsに記録しない
- `.env.local` はGit管理しない
- localStorageへURL / key / tokenを独自保存しない

GitHub Pagesでの静的運用候補:

| 候補 | 内容 | 注意 |
| --- | --- | --- |
| 公開可能なanon / publishable keyを静的JSへ入れる | no-build運用では単純 | 本当に公開可能なkeyだけに限定する。実値はdocsへ書かない |
| 設定ファイルを分離する | `supabaseAuthClient.js` などへ隔離 | 公開配信される点は同じ。差し替えとレビューはしやすい |
| ローカル確認用と公開用を分ける | devでは手入力、本番では公開可能設定 | 誤ってdev用secretを混ぜない運用が必要 |

推奨は、公開可能なanon / publishable keyだけを扱う設定を `supabaseAuthClient.js` に隔離し、実装前レビューで実値を確認する方式。ただし、この設計書には実値を記録しない。

## 8. エラー表示方針

想定エラー:

| 状態 | PL向け表示 |
| --- | --- |
| Supabase未接続 | アカウント機能を準備中です |
| セッション復元失敗 | ログイン状態を確認できませんでした |
| `public_profiles` 取得失敗 | 表示名を取得できませんでした |
| ネットワークエラー | 通信に失敗しました。時間をおいて再度お試しください |
| 未ログイン | ログインしていません |
| ログアウト失敗 | ログアウトできませんでした。再度お試しください |

エラー表示に出さないもの:

- 内部エラー全文
- Project URL
- anon key / publishable key
- token
- UUID全文
- email
- 内部SQL

開発確認で詳細が必要な場合も、ユーザーへ共有するログはURL / key / token / emailを伏せる。

## 11. ロールバック方針

問題が出た場合に戻せるよう、実装時は以下を分離する。

- 共通ヘッダーのアカウントアイコンDOM
- アカウントアイコン用CSS
- Auth状態取得JS
- マイページ本文描画JS

戻し方:

1. 共通ヘッダーからアカウントアイコン追加を外す
2. `accountNav.js` の読み込みを外す
3. `mypage.html` への導線を非表示にする
4. 静的サイト本体の閲覧、カレンダー、セッション詳細は維持する

AuthやコメントDBの状態変更を伴わない段階から入ることで、ロールバック時のDB復旧を不要にする。
