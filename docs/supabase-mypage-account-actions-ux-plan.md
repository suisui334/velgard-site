# Supabase マイページ全体のアカウント操作UX再設計書

この設計書は、Supabase Authを本番サイトへ実装する前に、`mypage.html` をアカウント操作の入口としてどう育てるかを整理するためのものです。

この工程では実装、HTML / JS / CSS変更、Supabase接続、Auth復元、ログイン / ログアウト、追加SQL実行は行いません。

## 1. 目的

- `mypage.html` をアカウント操作の入口として再設計する
- ログイン状態だけを大きく表示するカードは採用しない
- ログイン / ログアウト、申請状況、参加予定などの操作・情報と一体化して扱う
- Supabase接続前に、表示内容、段階実装、安全条件、ロールバック方針を固定する
- `session-detail.html` 本文中にログイン状態やログインフォームを常時表示しない方針を維持する

A-4aでは、`mypage.html` にログイン状態カードだけを置く案をローカルで検討したが、ユーザー確認後のUX方針として採用しないことにした。ログイン状態そのものは、ログアウトしない限り維持されるため、独立カードとして強調するよりも、必要な操作へ進む入口として見せる。

## 2. これまでの確認結果の反映

| 工程 | 確認済み内容 | マイページUXへの反映 |
| --- | --- | --- |
| F-3 | Authログイン状態、ログアウト、セッション復元、`public_profiles.display_name` 取得 | 本番では表示名中心に扱い、token / user_id全文 / emailは出さない |
| F-4 | 参加希望コメント投稿、公開コメント再読込、参加人数RPC再読込 | 将来の「自分の申請状況」表示候補にする |
| F-5 | GMによる `accepted` / `rejected` 変更、権限差 | GM向け操作はマイページではなく、基本は該当セッション詳細上の管理UIへ寄せる |
| F-6 | コメント編集、論理削除、必要時の `canceled` 化 | 自分のコメント履歴や申請状況の将来拡張候補にする |

マイページは、全操作を詰め込む管理画面ではなく、PL本人が自分に関係する状態と導線を確認する入口として始める。

## 3. 最小構成案

初期実装では、ログイン状態カード単体ではなく、ログイン前後で「何ができるか」を見せる。

| 状態 | 表示案 | 操作案 |
| --- | --- | --- |
| 未ログイン | マイページ / ログインすると参加申請状況などを確認できます | ログイン導線、トップへ戻る、CALENDARへ戻る |
| ログイン中 | マイページ / `display_name` / 参加申請一覧は今後対応 / 参加予定セッションは今後対応 | ログアウト導線、トップへ戻る、CALENDARへ戻る |
| 読み込み失敗 | マイページ / アカウント情報を確認できませんでした | 再読込案内、トップへ戻る、CALENDARへ戻る |
| 接続設定未構成 | マイページ / アカウント機能は準備中です | 既存の静的準備中表示を維持 |

最小版の本文例:

```text
マイページ
ログインすると、参加申請状況や参加予定セッションを確認できるようになります。
ログイン機能は準備中です。
トップへ戻る / CALENDARへ戻る
```

ログイン実装後の本文例:

```text
マイページ
ようこそ、表示名さん。
参加申請一覧と参加予定セッションは今後対応予定です。
ログアウト
```

## 4. ログイン / ログアウト導線

検討項目:

| 項目 | 方針 |
| --- | --- |
| ログインフォームをmypage内に置くか | 初期推奨。ACCOUNT導線の到着先で完結し、本文ページを汚さない |
| ログイン専用ページを作るか | 将来、認証UIが大きくなる場合に検討する |
| ログアウトボタンの場所 | ログイン中のマイページ内に置く |
| ログアウト後の遷移先 | 初期案は `mypage.html` に留まり、未ログイン向け表示へ切り替える |
| 未ログイン時にACCOUNTを押した場合 | ログイン案内とログイン導線を表示する |
| ログイン中にACCOUNTを押した場合 | 表示名、自分向け導線、ログアウト導線を表示する |

推奨案:

- ログイン / ログアウトは `mypage.html` 内で扱う
- 共通ヘッダーは静的な `ACCOUNT` 導線のまま始める
- `session-detail.html` 本文中にはログインフォームを置かない
- セッション詳細で投稿が必要な場合も、ログインしていなければマイページまたはログイン導線へ誘導する

## 5. 表示してよい情報 / 表示しない情報

表示してよい情報:

- `display_name`
- 自分の申請状態
- 自分の参加予定
- 簡潔なログイン案内
- 簡潔なエラー文

表示しない情報:

- `user_id` 全文
- `discord_user_id`
- email
- access token
- refresh token
- service_role
- secret key
- DB password
- 内部role
- Supabase Project URL
- anon key / publishable key

メール表示は原則出さない。必要になる場合も、本人向けの限定表示として別途検討する。

- Auth復元や `display_name` 表示は、マイページ内の操作導線と一体で扱う

## 7. Supabase接続情報の扱い

必須方針:

- service_role は絶対に使わない
- secret key / DB password は絶対に使わない
- Project URL / anon key / publishable key 実値をREADMEやdocsに記録しない
- `.env.local` はGit管理しない
- localStorageへURL / key / tokenを独自保存しない
- Supabase Auth標準のセッション永続化を使う場合は、保存先とログアウト時の挙動を実装前に確認する

GitHub Pages静的運用での候補:

| 案 | 内容 | 評価 |
| --- | --- | --- |
| 静的JSに公開可能keyを入れる | ビルドなしで単純 | 実値commitの可否判断が必要。docsには書かない |
| runtime configファイル分離 | 接続情報の責務を分ける | 推奨。設定差し替えとロールバックがしやすい |
| 未構成フォールバック先行 | 接続値なしで壊れない器を先に作る | 安全。M-2向き |

推奨は、M-2で未構成フォールバックを整え、M-3でruntime config分離方針を確定してからAuth復元へ進むこと。

## 8. エラー表示方針

| 状態 | 表示方針 |
| --- | --- |
| 接続設定未構成 | アカウント機能は準備中です |
| Supabase初期化失敗 | アカウント機能を読み込めませんでした |
| Authセッション復元失敗 | ログイン状態を確認できませんでした |
| ログイン失敗 | ログインできませんでした。入力内容を確認してください |
| ログアウト失敗 | ログアウトできませんでした。時間を置いて再度お試しください |
| public_profiles取得失敗 | 表示名を取得できませんでした |
| ネットワークエラー | 通信に失敗しました。時間を置いて再度お試しください |

PL向けには短く、人間向けに表示する。内部SQL、token、key、UUID全文、email、Project URLは画面にもconsoleにも出さない。

## 9. ロールバック方針

実装後に問題が出た場合は、以下の順で戻せるようにする。

1. Auth用JS読み込みを外す
2. `mypage.html` を静的準備中表示へ戻す
3. `assets/js/renderMypage.js` をA-3最小版へ戻す
4. runtime config読み込みを止める
5. 共通ヘッダーの静的 `ACCOUNT` 導線は必要に応じて維持する

この段階ではDB状態変更を伴わないため、Supabase側ロールバックは不要にする。

## 12. M-2/M-3 runtime config / 未構成フォールバック計画への分離

M-2/M-3の詳細は `docs/supabase-mypage-runtime-config-fallback-plan.md` に分離する。

Auth実装へ進む前に、実Project URL / anon key / publishable keyをGitHub、README、docs、チャットへ残さない運用と、接続設定が未構成でも `mypage.html` が壊れないフォールバックを先に整理する。

M-2/M-3最小実装では、ログイン状態カード単体ではなく、マイページ内の「アカウント機能」セクションとして未構成フォールバックを表示する。Supabase接続、Auth復元、ログイン / ログアウト、申請一覧表示はまだ扱わない。

## 13. M-4 実config運用方針レビューへの分離

実config運用方針とSupabase接続前レビューは `docs/supabase-mypage-runtime-config-deployment-review-plan.md` に分離する。

M-4では、GitHub Pages静的運用で公開可能keyをどう扱うか、repoに実値をcommitするか、接続前にRLS / public RPC / エラー・ログ方針をどう確認するかを整理する。`mypage.html` のアカウント操作UX自体は、ログイン状態単体カードではなく、ログイン / ログアウトや申請一覧と一体で扱う方針を維持する。

## 14. M-6 ログイン / ログアウト計画への分離

M-6のログイン / ログアウト最小実装計画は `docs/supabase-mypage-login-logout-plan.md` に分離する。

M-6では、`mypage.html` のアカウント機能セクション内にメールアドレス + パスワードログインとログアウトを最小導入する方針を整理する。ログイン状態カード単体には戻さず、email / user_id / tokenを表示しない。`display_name`、自分の申請一覧、参加予定セッション、`session-detail.html` 投稿統合は後続工程へ分ける。

## 15. M-7 一般サインアップ計画への分離

ユーザー方針として、アカウント登録方式はサイト上に誰でも登録できる一般サインアップフォームを置く方式に決定した。M-7の一般サインアップ実装計画は `docs/supabase-mypage-signup-plan.md` に分離する。

M-7では、`mypage.html` のアカウント機能セクション内でメールアドレス + パスワード + パスワード確認による新規登録、Supabase Auth `signUp`、登録成功時の短い案内、ログインフォームとの切り替えを扱う前提を整理する。既存の段階案でM-7候補だった `display_name` 表示 / 登録は後続工程へ送る。
